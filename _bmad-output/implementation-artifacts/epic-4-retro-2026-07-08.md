# Epic 4 Retrospective: Source-Grounded AI Answers And Trust Signals

## Epic Review

### Outcome

Epic 4 is complete. Its seven stories make AI Ask answers source-grounded, state-aware, and inspectable without exposing raw or operator-only material. Retrieval now rechecks current knowledge state rather than trusting historical approval or index state; indexed projections are versioned and fail closed; policy controls how community, conditional, uncertain, and conflicted material may be used; web fallback and provenance are persisted; and traveler-facing trust details render only safe metadata.

### Completed Stories

| Story | Status | Result |
| --- | --- | --- |
| 4.1 Migrate Retrieval to State-Aware Active Knowledge | done | Added current-state eligibility evaluation and contextual, caveat-only, or excluded use policies. |
| 4.2 Index Current AI-First Knowledge Versions | done | Added versioned, idempotent indexing with stale and ineligible projection protections. |
| 4.3 Assemble State-Aware Knowledge Source Bundles | done | Added bounded traveler-safe source bundles for retrieval and prompt construction. |
| 4.4 Enforce Community, Conditional, and Conflict Answer Policy | done | Added answer-policy constraints and required caveats for changing, uncertain, conditional, and conflicted information. |
| 4.5 Update Search Fallback and Provenance for AI-First States | done | Added safe web fallback, persisted retrieval decisions, and row-per-source provenance. |
| 4.6 Render State-Aware Traveler Trust Details | done | Added safe traveler trust details and inspection behavior backed by stored provenance. |
| 4.7 Verify AI-First Retrieval and Answer Safety | done | Added cross-story regression coverage for retrieval eligibility, policy, provenance, worker safety, and no-raw-data boundaries. |

### What Went Well

- Retrieval eligibility became a current-state decision. Search ranking and historical approval fields cannot override publication, evidence, source, verification, review, or privacy constraints.
- The three explicit use policies make uncertainty visible in the contract: safe facts can be contextual, verification-required information is caveat-only, and incomplete or conflicted material is excluded.
- Prompt construction and traveler UI share a bounded source/provenance model instead of deriving trust claims from assistant response text.
- Code-review repairs closed safety gaps around Facebook evidence filtering, credential-bearing external URLs, and deterministic web-fallback guidance before Epic closure.
- The Epic 4 verification suite consolidated state, source, policy, web fallback, provenance, and worker timing concerns into regression coverage.

### Challenges

- The revised Epic 4 superseded an earlier intake-and-approval Epic 4. Its historical retrospective became misleading and has now been replaced with this closure record.
- Indexing is necessarily asynchronous, so retrieval must continue to recheck current owner rows and source/evidence state even after a projection is indexed.
- Traveler trust UI needs sufficiently useful source metadata while preserving the boundary against raw source text, private data, operator notes, provider payloads, and unsafe links.
- DB-backed suites require serial execution where reset and migration state are shared. This is documented in the canonical README guidance but remains an operational constraint.

### Recurring Review Findings

- Authorization and current-state checks must precede provider calls, persistence, prompts, and streamed output.
- Raw, private, operator-only, and unsafe evidence must not enter source bundles, provenance snapshots, traveler DTOs, logs, or prompts.
- A lexical projection is only a candidate. Current card, evidence, capture, source, review, publication, and verification checks determine traveler eligibility.
- Conflicted claims cannot become itinerary premises; conditional, freshness-sensitive, and verification-required claims need explicit caveats and confirmation guidance.
- Provenance, annotations, and trust renderers must use stored safe records, never parse free-form answer text.

### Testing And Quality Lessons

- Focused tests should include stale projections, source withdrawal, archived or superseded cards, missing sources, operator-only evidence, independent-support requirements, conflicts, and verification-required states.
- Web fallback tests must cover low confidence, provider failure, unsafe URLs, and the absence of supported internal knowledge; each path must give verification guidance rather than invent current facts.
- Run DB-backed suites sequentially with `DATABASE_URL_TEST`; do not combine shared-reset suites with unsafe parallel execution.
- The final Epic 4 regression checks passed `pnpm lint`, `pnpm typecheck`, `pnpm build`, and focused retrieval/answer-context tests. Lint retains three documented pre-existing unused-variable warnings in `tests/knowledge-search.test.ts`.
