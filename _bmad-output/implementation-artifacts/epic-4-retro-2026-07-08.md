# Epic 4 Retrospective: AI-Assisted Knowledge Intake And Approval

## Epic Review

### Outcome

Epic 4 is complete. All 9 planned stories are marked `done` and together deliver the operator-side knowledge pipeline:

1. Submit source material safely.
2. Extract draft knowledge cards with AI.
3. Review, edit, reject, and approve drafts.
4. Suggest create/update/no-action outcomes from source URLs.
5. Batch seed source URLs.
6. Preserve source and confidence metadata on approved cards.
7. Make approved knowledge searchable by the server.
8. Track progress toward 100 approved Hanoi-to-HCMC corridor items.

Epic 4 intentionally stopped short of traveler-facing retrieval, web search fallback, answer provenance UI, and vector embeddings. Those remain Epic 5 concerns.

### What Went Well

- Raw-source privacy became a durable invariant. Epic 4 repeatedly reinforced the split between safe source metadata and operator-only raw source material.
- Human approval stayed the hard gate. Drafts, suggestions, duplicate/no-action traces, and search documents do not become traveler-usable knowledge until explicit approval criteria are met.
- Operator workflow became coherent end to end. The admin path now supports intake, extraction, review, approval, inspection, search preparation, and seed progress tracking.
- Code review found high-value issues before closure. Review loops caught authorization order, raw metadata leakage, stale search documents, confidence clamping, migration metadata drift, lifecycle race guards, and DTO overexposure.
- Epic 4 prepared Epic 5 without prematurely implementing it. Story 4.8 created a server-owned approved-knowledge search boundary while deferring AI Ask retrieval, answer provenance, web search, and vector embeddings.

### Challenges

- Safe metadata was harder than simple table separation. Labels, publishers, summaries, tags, practical details, audit summaries, and DTOs can all become leakage paths if not constrained.
- Lifecycle semantics required repeated tightening. Draft, needs-review, approved, rejected, archived, duplicate, no-action, and stale-search states all needed explicit handling.
- Migration metadata drift appeared in multiple schema-heavy stories. Drizzle snapshots and journal alignment are now a recurring quality path.
- Parallel DB-backed test execution caused false failures. Several focused suites passed sequentially but interfered when run together against shared state or while generated Next.js types changed.
- Some necessary MVP decisions were deferred: screenshot extraction, URL fetching, automatic indexing/backfill, source label/publisher hardening, vector retrieval, and canonical corridor taxonomy.

### Recurring Review Findings

- Authorization must happen before side effects: parsing, provider calls, database writes, usage writes, and audit writes.
- Raw/operator-only material must never leak through safe fields, UI, audit, search documents, DTOs, source bundles, or future traveler prompts.
- Retrieval eligibility cannot trust one status flag. It needs approved status, `needsReview=false`, active source linkage, non-archived state, and current indexed content.
- Concurrency/idempotency matters around source extraction, suggestions, approval, batch status persistence, and stale indexing.
- Confidence and provenance should come from normalized source rows and lifecycle state, not free-text card fields.
- DB migration verification and test isolation need stricter operating rules as the schema grows.

### Testing And Quality Lessons

- Focused story tests were strong around auth-before-side-effects, raw material absence, DTO shape, lifecycle exclusion, stale index behavior, migration metadata, and safe audit/usage logging.
- Full-suite and adjacent regression checks were useful but became sensitive to shared DB state and generated build/type artifacts.
- Epic 5 needs stronger integration tests across retrieval, source bundle assembly, assistant message persistence, provenance persistence, and traveler-facing rendering.
- Any retrieval tests should prove draft, rejected, archived, needs-review, source-orphaned, stale-search, and raw-source rows never appear in traveler source bundles.

## Next Epic Preparation

### Epic 5 Preview

Epic 5 is already in progress. Story 5.0 is complete and added AI Gateway model/pricing management. The remaining Epic 5 work covers approved knowledge retrieval, context-priority source bundles, web search fallback, retrieval decisions, answer provenance, source/confidence UI, uncertainty warnings, web search validation, and usage events.

Epic 5 depends directly on Epic 4's approved knowledge, safe source metadata, lifecycle gates, and search document boundary.

### Readiness Risks For Epic 5

- The context digest/source-bundle policy is still unresolved. Epic 5 needs explicit max facts, max characters, allowed fields, conflict handling, prompt-injection treatment, and priority ordering.
- Epic 4 delivered PostgreSQL search documents, not vector embeddings. Epic 5 must intentionally choose whether to start with text search or add pgvector/embedding retrieval now.
- Search indexing/backfill is not yet automatically orchestrated. Retrieval will underperform or miss approved cards if indexing is manual or inconsistent.
- Source label and publisher fields need a traveler-facing safety decision before Story 5.6 renders source/confidence metadata.
- Answer provenance persistence is more complex than approved-card provenance inspection. Epic 5 must persist retrieval decision and row-per-source provenance with assistant messages without parsing answer text.
- Web search provider quality remains undecided. Story 5.8 needs provider/mechanism validation for Vietnamese corridor quality, official-source preference, URL/title/snippet/date availability, cost, limits, and failure behavior.
- Retrieval eligibility must remain fail-closed by joining search documents back to current approved cards and source links.

### Significant Discovery Assessment

Epic 4 did not reveal a product-breaking mismatch, but it did reveal implementation constraints that should be reflected before continuing deeper Epic 5 retrieval work.

Epic 5 planning should be clarified because Epic 4 delivered approved-only PostgreSQL search documents rather than embeddings, and indexing/backfill orchestration remains incomplete.

Architecture should be updated or annotated to capture these invariants:

- PostgreSQL search documents are the current approved-only search boundary unless Epic 5 explicitly adds pgvector.
- Raw source material remains operator-only and must never enter traveler source bundles.
- Linked normalized source rows, not free-text card fields, are the provenance source.
- Retrieval requires status, needs-review, source-link, and stale-index checks.
- Any future embeddings must become stale or disabled when card/source lifecycle state changes.

PRD update is not required unless the product expectation is that semantic/vector retrieval must ship in the public MVP. If PostgreSQL text search is acceptable as an MVP retrieval step, this is an architecture/story clarification rather than a product requirement change.
