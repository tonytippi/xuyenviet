### What Went Well

1. **Epic 1 retro action was actually closed.** Story 2.0 added Vitest, a separate `DATABASE_URL_TEST`, Drizzle migration setup, no-network guards, and retroactive coverage for auth gate, admin roles, audited mutations, and production env guards.
2. **The team kept fail-closed behavior under pressure.** Invalid or unauthenticated AI Ask submissions still create no conversations, messages, usage events, or provider calls.
3. **Ownership boundaries improved with each review.** Story 2.2 added database-level conversation/message ownership integrity. Story 2.5 strengthened loaded-message reads with user ID filtering as defense in depth.
4. **AI generation entered through the right seam.** Story 2.3 used a server-only OpenAI-compatible AI Gateway adapter, recorded usage events for success/failure, and avoided direct OpenAI calls or raw provider payload storage.
5. **Assistant messages became the durable UI source of truth.** Stories 2.3 through 2.6 consistently preserved the rule that displayed assistant content comes from persisted server results, not client-only fabricated state.
6. **The answer-format contract stayed honest.** Story 2.4 improved Vietnamese structured answers while explicitly avoiding fake citations, fake source chips, provenance rows, retrieval rows, and parsed-source trust.
7. **Failure and waiting states became usable.** Story 2.6 added in-thread pending/progress/failure presentation, duplicate-submit guarding for the mounted composer, and failed-turn notices that survive refresh/remount.
8. **Verification discipline held.** Each story recorded `pnpm test:run`, `pnpm lint`, `pnpm typecheck`, and `pnpm build`; typecheck/build race failures were identified as `.next/types` regeneration races and rerun sequentially.

### Challenges

1. **Testing infrastructure created useful friction.** Story 2.0 required real test database safety guards, migration setup, `.env` precedence handling, and network-call blocking. This was necessary, but it increased setup sensitivity around local PostgreSQL and Node 20.19+.
2. **Composer behavior is still not covered by a browser interaction harness.** Duplicate-submit and client interaction behavior are covered through source/static/server tests, not real browser or jsdom interaction tests. This remained a deliberate scope boundary.
3. **Prompt behavior remains partly contract-based.** Story 2.4's structured format is prompt-driven rather than provider-enforced structured output. Good enough for Epic 2, but later provenance/retrieval work must not rely on parsing answer text for trusted metadata.
4. **Cross-tab concurrency is unresolved.** Story 2.5 found that two tabs can submit to the same conversation at once, causing both provider prompts to read stale prior history before either sees the other in-flight turn.
5. **Usage instrumentation is minimal by design.** Story 2.3 records durable usage events, but richer source bundle, provenance, cost, retrieval-decision, and model-evaluation behavior remains future work.
6. **Epic 3 will put more weight on existing conversation tables.** Epic 2 created conversations/messages as the current chat-session substrate; Epic 3 now needs history, session management, trip projects, context extraction, correction, and deletion semantics without breaking existing rows.

### Key Insights

- **Retrospectives need teeth.** The Epic 1 retro created Story 2.0; Epic 2 proved that action-item insertion can materially reduce risk before feature work builds on a weak foundation.
- **Persisted source-of-truth is the right invariant.** The strongest Epic 2 implementation pattern was simple: server validates, persists, calls provider when allowed, persists assistant output only on success, and UI renders persisted content.
- **Concurrency becomes real once chat is continuous.** A single-message MVP can ignore ordering more easily; a multi-turn planning assistant cannot. Epic 3 should make serialization/idempotency an explicit design decision rather than a hidden edge case.
- **Source/provenance restraint prevented future rework.** By refusing fake source UI in Epic 2, the project preserved room for Epic 5's real retrieval/provenance pipeline.
- **Static checks plus integration tests are now the baseline.** Story 2.0 changed the engineering floor: future persistence, ownership, AI provider, deletion, and failure-state stories should add tests, not only lint/typecheck/build.

### Team Agreements

- Keep server-side integration tests mandatory for stories that add persistence, ownership checks, AI provider calls, deletion behavior, or failure-state contracts.
- Do not render source/confidence/provenance UI from parsed model text. Wait for real retrieval/provenance rows.
- Treat conversations/messages as user-owned data with server-derived ownership. Do not accept caller-supplied owner IDs for protected reads or writes.
- Before Epic 3 context extraction and answer-context reuse, decide whether conversation continuation needs transaction-level serialization, optimistic conflict detection, idempotency keys, or a queue/lock pattern.
- When `pnpm typecheck` and `pnpm build` interact with `.next/types`, run them sequentially for reliable verification reporting.
- Keep deferred work actionable: every deferred item needs source, risk, and a likely resolution point.

## Part 2 — Next Epic Preparation

### Epic 3: Chat Sessions And Trip Projects

Epic 3 lets travelers organize planning across chat sessions and trip projects, reuse relevant context within each scope, correct trip details through chat, and delete chats or trip projects they own.

Planned stories:

- Story 3.1 — Manage Chat Sessions
- Story 3.2 — Create Trip Projects
- Story 3.3 — Extract Chat And Trip Context
- Story 3.4 — Use Chat Or Trip Context In Answers
- Story 3.5 — Correct Trip Details Through Chat
- Story 3.6 — Delete Chat Sessions
- Story 3.7 — Delete Trip Projects

### Dependencies On Epic 2

- Story 3.1 builds directly on Epic 2 conversations/messages and continuation behavior.
- Story 3.2 introduces project ownership and links project scope to chat sessions.
- Stories 3.3 through 3.5 depend on clear separation between chat-session context and trip-project context.
- Stories 3.6 and 3.7 depend on deletion behavior for user-owned chat/project-derived content before that content becomes retrievable.
- Story 2.5's concurrent continuation deferral affects the reliability of chat history used by 3.1, 3.3, and 3.4.

### Preparation Needs

1. **Conversation/session naming decision.** Epic 2 already has `conversations`; Story 3.1 should decide whether those rows are the chat sessions, whether a title/summary/listing field is added, or whether a separate `chat_sessions` abstraction is needed.
2. **Concurrency strategy.** Same-conversation concurrent sends need an explicit behavior before the app relies on session history as context.
3. **Deletion model before new retrievable content.** Stories 3.6 and 3.7 require a clear remove/disable behavior for messages, extracted context, and future embeddings.
4. **Context data boundaries.** Story 3.3 must distinguish chat-scoped temporary details from durable trip-project context and reject clearly disallowed sensitive data.
5. **Testing plan.** Epic 3 tests should cover ownership, cross-user denial, deletion effects, context scope, correction ambiguity, and no leakage across sessions/projects.

### Readiness Verdict

- **Epic 2 is complete enough to close.** The public AI Ask loop now supports authenticated Vietnamese conversation, first answers, structured answer formatting, continuation, usage recording, and recoverable failure UI.
- **Epic 3 may start with Story 3.1** if the story explicitly addresses whether current `conversations` are the chat-session model and how the open concurrency item is handled.
- **Do not start context extraction/use stories blindly.** Story 3.3 and 3.4 should wait until session/project ownership, deletion implications, and concurrency behavior are clear.
