### What Went Well

1. **Epic 3 delivered the personalization substrate.** Travelers can now manage multiple chat sessions, create trip projects, extract travel context, use remembered context in later answers, correct remembered details, and delete chats/projects they own.
2. **Ownership boundaries held.** All implemented reads and mutations resolve the authenticated user server-side and avoid exposing cross-user conversations, projects, context rows, or destructive actions.
3. **The team preserved the Chat/Trips ownership model.** Conversations, messages, trip projects, chat context, answer-context reads, and user-owned deletion stayed inside the Chat/Trips module instead of leaking generic cross-module data helpers.
4. **Review loops caught real issues before closure.** Story 3.2 fixed FK delete behavior, project/conversation mismatch rendering, project chat leakage, and over-detailed audit summaries. Story 3.3 fixed sensitive-data filtering, same-user project/context integrity, async extraction behavior, and transactional consistency.
5. **Context and deletion contracts were clarified before retrieval.** Stories 3.6 and 3.7 now define how chat deletion, project deletion, context cascades, usage metadata, and linked-chat detachment behave before Epic 5 introduces retrievable provenance and embeddings.
6. **Verification discipline stayed strong.** The project moved from Epic 2's test baseline into broader Epic 3 coverage for sessions, projects, extraction, answer context, corrections, chat deletion, and project deletion.
7. **The team avoided fake source/provenance shortcuts.** Epic 3 used chat/trip context as personalization input only and did not invent source/confidence UI before real retrieval/provenance rows exist.

### Challenges

1. **User context memory is the riskiest Epic 3 area.** Tony identified the central risk: the system must store enough context to be useful while keeping it small, scoped, safe, and understandable.
2. **Story 3.2 and Story 3.3 required significant review hardening.** Project links, context persistence, sensitive-data rejection, malformed model output, usage/audit recording, and DB integrity all needed deeper patches than the initial story execution captured.
3. **Story 3.4's prompt approach needs clearer product/architecture documentation.** Remembered chat/trip context is currently injected into the answer prompt's system/developer context area as data-only application context. This is necessary for the model to use backend memory, but it remains a soft prompt boundary.
4. **Prompt-injection risk is bounded, not eliminated.** User-derived context values are serialized as data and owner-scoped, but determined self-targeted prompt injection through remembered context remains a residual risk.
5. **Context freshness is intentionally one turn stale.** Extraction runs after the answer stream via `after()`, so current-turn context and corrections generally influence later answers rather than the same streamed answer.
6. **Deletion is correct for current tables but future storage needs more rules.** External object-storage cleanup is deferred before non-null image storage keys are introduced; future embeddings must adopt the same owner/status/deletion contract.

### Tony's Key Feedback

Tony identified the most important risk as context management:

> The most risky is how we manage the context from users, make it small but enough, store 3.2 and 3.3. For 3.4, I still don't understand why we inject into the system prompt.

Retrospective interpretation:

- The concern is valid and should drive follow-up work.
- Epic 3 made the MVP memory mechanism functional, but it needs a documented context digest policy before the answer pipeline adds approved knowledge and web fallback.
- The current Story 3.4 approach is acceptable for MVP because the model cannot use database memory unless the server passes a bounded context digest into the model request.
- The context digest must be treated as application-provided data, not user instructions or trusted provenance.
- Retrieval/tool-calling strategy for large backend knowledge was intentionally deferred and captured in `_bmad-output/implementation-artifacts/deferred-work.md` under "Future brainstorm: retrieval and tool-calling strategy".

### Key Insights

- **Memory is product behavior, not just persistence.** The user experience depends on what the app remembers, what it ignores, when it asks clarification, and when it deletes or stops using context.
- **Small context is a feature.** A compact digest is safer and more useful than dumping all stored context into the model request.
- **Scope is the main safety primitive.** Conversation context, trip-project context, approved knowledge, and web fallback need explicit boundaries and priority rules.
- **Prompt use is necessary but insufficient.** Passing remembered context to the model is required for personalization, but the system must rely on server-side scoping, allowlists, validation, size caps, and provenance rows rather than model obedience alone.
- **Epic 5 should not parse answers for truth.** Stored provenance must come from the server-selected source bundle, not from the model's final text.

## Part 2 - Next Epic Preparation

### Epic 4: AI-Assisted Knowledge Intake And Approval

Epic 4 lets operators submit travel links, copied content, Facebook/community material, and screenshots; AI prepares structured knowledge drafts; operators review/edit/approve useful items; and approved knowledge becomes available for later traveler-answer retrieval.

Planned stories:

- Story 4.1 - Submit Travel Source For AI Reading
- Story 4.2 - AI Extracts Knowledge Drafts From Source
- Story 4.3 - Review And Edit AI-Prepared Drafts
- Story 4.4 - AI Suggests Create Or Update From Source URL
- Story 4.5 - Batch Seed Source URL Intake
- Story 4.6 - Approve Knowledge For Retrieval
- Story 4.7 - Preserve Source And Confidence In Approved Knowledge
- Story 4.8 - Make Approved Knowledge Searchable By AI
- Story 4.9 - Track 100 Approved Corridor Items

### Dependencies On Epic 3

- Epic 4 should reuse the same server-side ownership and role-check discipline, but with operator/admin permissions instead of traveler ownership.
- Raw source material and AI-extracted drafts must remain operator-only until human approval.
- Traveler answer prompts must not receive raw source material, copied Facebook text, screenshots, provider payloads, or unapproved draft facts.
- Approved knowledge should later feed Epic 5 as a compact source bundle with stable source IDs and provenance metadata.
- Deletion/status rules from Epic 3 should influence any future retrievable content or embeddings: retrieval must join back to current owner/status rows and avoid stale/deleted content.

### Preparation Needs

1. **Context digest policy.** Define max facts, max characters, allowed fields, conflict handling, and which stored context fields are allowed into model prompts.
2. **Raw-source privacy policy.** Epic 4 must clearly separate raw operator-only submitted material from traveler-safe source metadata.
3. **Knowledge-card source model.** Ensure `knowledge_card_sources` and source metadata are the future source of traveler-facing provenance, not free-text card fields.
4. **Approval lifecycle invariants.** Draft and unapproved knowledge must not become retrievable by normal traveler AI Ask.
5. **Future retrieval strategy.** Keep MVP server-controlled retrieval as the default; revisit model tool calls later for agentic workflows only after auth, role, approval, privacy, budget, and provenance constraints are explicit.

### Readiness Verdict

- **Epic 3 is complete enough to close.** The traveler memory/session/project/delete foundation is in place with strong test coverage and no current blockers.
- **Do not ignore the context-memory risk.** Add a context digest policy action before retrieval/provenance work expands the answer pipeline.
- **Epic 4 may start with Story 4.1** if it keeps raw source material operator-only and avoids feeding unapproved backend knowledge into traveler AI Ask.

## Team Agreements

- Treat chat/trip context as personalization input, not trusted provenance.
- Keep remembered context compact, allowlisted, owner-scoped, and data-only in model requests.
- Do not send raw or unapproved operator source material to traveler-facing AI Ask prompts.
- Persist provenance from server-selected source bundles, not from parsed model output.
- Keep deletion/status behavior defined before introducing any new retrievable table or embedding table.
