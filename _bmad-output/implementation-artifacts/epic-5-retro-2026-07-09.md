# Epic 5 Retrospective: Grounded Retrieval, Web Search, And Provenance

## Epic Review

### Outcome

Epic 5 is complete. All 10 planned stories are marked `done` and together deliver the traveler-facing grounding pipeline:

1. Manage callable AI Gateway model records and pricing metadata.
2. Retrieve approved XuyenViet knowledge for AI Ask.
3. Assemble a priority-ordered source bundle from trip context, chat context, approved knowledge, web fallback, and general reasoning.
4. Decide when web fallback is needed without fabricating web claims.
5. Capture web fallback sources through a provider adapter.
6. Persist retrieval decisions and row-per-source answer provenance.
7. Render source and confidence sections from stored provenance.
8. Add uncertainty and freshness warnings for changing or unverified details.
9. Validate web search fallback quality against Vietnamese corridor needs.
10. Standardize AI usage event recording for Gateway and search-provider calls.

Epic 5 intentionally kept web results external and unverified. It did not convert search results into approved XuyenViet knowledge, add credit/billing behavior, or make pgvector retrieval a public-MVP dependency.

### What Went Well

- The architecture held. Epic 5 followed the modular monolith, server-owned orchestration, approved-only retrieval, adapter-based web search, and persisted-provenance decisions from the architecture spine.
- Source-bundle assembly became an explicit contract instead of prompt string accumulation. This made ordering, category labels, bounds, and prompt-injection handling reviewable.
- Web search stayed behind a provider adapter and remained external/unverified. Tavily can be replaced without changing source display, grounding, or warning semantics.
- Provenance moved from intent to durable product state. Assistant answers now have retrieval decisions and row-per-source provenance that UI and future evaluation can inspect.
- Source/confidence rendering correctly depends on stored provenance rather than model-generated citations or parsed answer text.
- Usage telemetry stayed operational, not financial. The implementation records cost/usage metadata where available without creating balances, credits, rewards, or payment obligations.
- Code review caught high-value safety issues before closure: prompt injection paths, spoofed official sources, unsafe URL rendering, query minimization, stale/freshness warning gaps, and usage event overexposure.

### Challenges

- Prompt-only behavior was not enough for freshness and uncertainty. Story 5.7 needed deterministic warning insertion so required Vietnamese warnings are streamed and persisted even if the model omits them.
- Web trust classification was subtle. Official/provider status had to be narrowed to safe parsed host checks instead of URL/title substring matching.
- Privacy boundaries expanded beyond raw source material. Web queries, usage events, provider metadata, snippets, prompts, answers, and source snapshots all needed explicit safe-content limits.
- Persistence and retry semantics became more complex once assistant messages, retrieval decisions, provenance, and usage events were tied together.
- DB-backed tests remained sensitive to parallel execution and generated Next.js type artifacts. Several focused suites are reliable sequentially but interfere when run together against shared state.
- Web quality validation is still fixture-backed. It proves evaluator behavior and documents provider risk, but it does not replace live monitoring of Tavily quality, cost, rate limits, and failure modes.

### Recurring Review Findings

- Treat retrieved data and web results as data, not instructions. Delimiters, JSON-style value encoding, bounded sections, and explicit prompt framing were needed repeatedly.
- Fail closed on eligibility and safety. Draft, rejected, archived, needs-review, raw-source, stale, unsafe URL, and unverified web paths must stay excluded or visibly labeled.
- Do not claim source confidence from the answer text. Provenance and confidence must come from stored rows and normalized source metadata.
- Web fallback must degrade to warnings, not invention. If web search is unavailable, failed, low quality, or not yet performed, the assistant must not imply current verification.
- Usage records must not become content storage. They can store provider/model/status/token/cost metadata, but not raw prompts, answers, queries, result snippets, or provider payloads.
- Auth, validation, and safe source-bundle assembly must complete before provider calls or user-visible generation.

### Testing And Quality Lessons

- Epic 5 consistently used targeted Vitest suites plus `pnpm lint`, `pnpm typecheck`, and `pnpm build` for story verification.
- The strongest regression coverage lives around `tests/answer-context.test.ts`, `tests/ai-ask-shell.test.ts`, `tests/ai-ask-sessions.test.ts`, `tests/web-search-adapter.test.ts`, `tests/web-search-quality.test.ts`, `tests/ai-usage-events.test.ts`, and `tests/ai-models.test.ts`.
- Tests now cover source-bundle ordering, web fallback decisions, prompt contracts, provenance rendering, unsafe URL filtering, freshness warning insertion, web adapter normalization, usage metadata safety, and model catalog selection.
- The remaining quality gap is not unit coverage; it is live public-MVP evidence. Epic 6 should use feedback and evaluation runs to prove answers are useful, family-aware, grounded, and better than generic AI.

## Next Epic Preparation

### Epic 6 Preview

Epic 6 is `Family-Aware Planning And Public MVP Quality Loop`. It adds child/family need detection, family-aware driving and stop recommendations, family-aware activity suitability notes, answer usefulness feedback, standard evaluation prompt runs, and a quality dashboard.

Epic 6 depends directly on Epic 5's source bundle, retrieval decisions, answer provenance, source/confidence UI, freshness warnings, web fallback quality report, and usage telemetry.

### Readiness Risks For Epic 6

- Family-aware answers will depend on the same source-bundle and warning pipeline. If family-specific details come from web/community sources, they must remain external/unverified until approved.
- Evaluation and feedback should inspect stored provenance and retrieval decisions rather than re-inferring grounding from answer text.
- The PRD's quality rubric includes family-awareness, source grounding, uncertainty handling, practical specificity, Vietnamese clarity, and user-context use. Epic 6 stories should keep those dimensions visible from the first story.
- Family context can involve children, but the privacy boundary remains travel-relevant facts only. Do not collect child full names, identity documents, medical details, exact home addresses, or other sensitive personal data.
- The usefulness feedback and quality dashboard should not accidentally become a billing, ranking, reward, or credit feature.

### Significant Discovery Assessment

Epic 5 did not reveal a product-breaking mismatch. The PRD and architecture remain directionally sound.

No PRD update is required before Epic 6. Architecture already reflects the key Epic 5 decisions: PostgreSQL-owned retrieval state, adapter-based Tavily seed provider, persisted answer provenance, managed AI Gateway models, and usage events as telemetry rather than billing.

The main planning implication is that Epic 6 should treat Epic 5's provenance tables and retrieval decisions as the source of truth for answer-quality measurement. Evaluation and dashboards should not parse answer text to determine grounding.
