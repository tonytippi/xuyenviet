- (decision) Adopt modular monolith on Next.js App Router for MVP; binds UI, API routes, admin, and AI orchestration into one deployable; prevents chat/admin/retrieval features from choosing incompatible service boundaries.
- (decision) Use PostgreSQL as system of record with pgvector for approved knowledge embeddings and memory embeddings; binds retrieval, provenance, deletion, and approval state to one owned data plane; prevents provider-hosted vector stores from becoming hidden source-of-truth.
- (decision) Use Drizzle for schema, migrations, and typed DB access; binds database evolution through code-reviewed migrations; prevents ad hoc SQL/schema drift across chat, admin, and eval work.
- (decision) Use Auth.js Google OAuth plus email allowlist and role table for private beta/admin; binds access decisions server-side; prevents client-only gating or separate admin auth.
- (decision) Model core domains as Auth, Chat, Memory, Knowledge, Retrieval, Search, AI Orchestration, Admin, Feedback/Eval, and Audit; dependencies flow inward through server modules, not direct cross-feature UI/database access; prevents circular ownership of user memory, cards, sources, and answer provenance.
- (decision) All mutations of memory, knowledge cards, source records, approvals, feedback, and deletion requests go through server actions/route handlers with authenticated user and audit context; prevents client-side writes and unaudited operator changes.
- (decision) Knowledge card lifecycle is draft -> approved -> archived; only approved cards can be retrieved for user answers; source material remains operator-only; prevents raw/unreviewed/Facebook-derived content from leaking into normal AI grounding.
- (decision) Retrieval pipeline order is memory + current trip context, approved knowledge hybrid retrieval, web search fallback, then general model reasoning; binds PRD priority into one orchestrator; prevents feature teams from bypassing source/confidence rules.
- (decision) Web search fallback uses a search adapter interface with Tavily as MVP seed provider; adapter output must normalize title, url, snippet/content, score, checkedAt, sourceType=external, confidence=unverified; prevents provider lock-in and missing provenance.
- (decision) OpenAI is used through a provider adapter for chat, extraction, embeddings, and evaluation prompts; all prompts must pass explicit source bundles and require structured provenance output; prevents model calls from inventing source labels or writing memory directly.
- (decision) Answer provenance is first-class data: every assistant response stores categories used, knowledge card IDs, memory IDs, web result IDs, model name, and prompt/eval metadata; prevents UI-only citations that cannot be audited.
- (decision) Memory is split into long-term user_memory and conversation trip_context; extraction proposes candidate memory updates and the memory module validates allowed fields before persistence; prevents sensitive data capture and current-trip facts polluting persistent profile.
- (decision) Deletion requests create a tracked privacy request; fulfillment deletes profile memory and derived memory embeddings and records tombstones; conversation transcript deletion remains admin-mediated during beta; prevents partial silent deletion and re-embedding deleted facts.
- (decision) Private beta operations run in dev, staging, and production environments with separate databases and secrets; production access requires allowlisted Google account and role check; prevents test data, beta users, and admin rights from mixing.
- (decision) Use hosted serverless-friendly Postgres with pgvector and a serverless Next.js deployment seed; operational owner must confirm provider before implementation; prevents architecture from relying on unmanaged local infrastructure.
- (decision) Streaming chat response is allowed only after retrieval/search context and provenance ledger are assembled; long extraction/embedding jobs may run as background tasks with status; prevents partial answers that cannot show required source/confidence section.
